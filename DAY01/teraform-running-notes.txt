count meta arguement:
When to Use count
Use count when:
You need multiple identical resources.
The only difference between them is usually an index or a small value.
Examples:
3 EC2 instances
5 local files
10 S3 buckets (with different names)
4 IAM users

Using count.index
-----------------
Usually, you don't want all three resources to have the same filename. You can use the built-in variable count.index.
we can create multiple instances of a resource or a module with single block.
for example we want to create two instaces, we copy the same resource block and duplicate the block and change alias name to "main1" to "main2". it is very difficult to do this for more instances.

lifecycle meta argument:
------------------------
1. create_before_destory
2. prevent_destroy
3. ignore_changes

IQ: when did you use life cycle meta argument explain

1.create_before_destroy:
create an ec2 instance by using tf file. it create an ec2 instance in mumbai region.
check in mumbai region.
for example i want to change something like changing the instance name from "main1" to "main2"  or ami id and apply the terrafomr file, first instance will be terminated and new instance will be created.
so application down time will be there as first instance is destroyed first and then second instance is created. 
so if new instance is created first and then old instance destroyed there will be no down time.

life cycle {
 create_before_destroy = true
}


2.prevent destroy:
------------------
if some one have the access to cli and by mistake if some one applied terraform destroy then application will be stopped.
so to avoid this we can give prevent destroy meta argument. 

prevent destroy= true

3.ignore changes:
---------------
suppose we edited a name tag in AWS UI but in local terrafrom state file tag is different. 
if we apply the code the in local the tag will become normal in UI as given in the local 

ignore_changes = [ tags ]
we if reapply the code the tag will be remain same in UI.

What is a Ternary Operator?
A ternary operator is a short way of writing an if-else condition.
The syntax is:
condition ? true_value : false_value
It means:
If the condition is true, use the first value; otherwise, use the second value.
Example 1
variable "create_instance" {
  default = true
}
Use it in count:
count = var.create_instance ? 1 : 0
This means:
If:
create_instance = true
Then:
count = 1
Terraform creates:
One EC2 instance
If:
create_instance = false
Then:
count = 0
Terraform creates:
Nothing

count = var.environment == "prod" ? 3 : 1
Meaning:
If:

1 EC2 instance

  default = false

Why Use a Ternary Operator?
Without it, you'd need separate Terraform configurations or manually edit count every time you switch environments.
With a ternary operator, one configuration can adapt automatically based on a variable.

I consider for_each one of the most important Terraform concepts because it is used heavily in real DevOps projects.
Why Do We Need for_each?
Suppose you want to create three files:
Suresh.txt
Ravi.txt
Krishna.txt
With count, you would do:
variable "names" {
  default = ["Suresh", "Ravi", "Krishna"]
}
resource "local_file" "students" {
  count = length(var.names)
  filename = "${var.names[count.index]}.txt"
  content = "Hello"
}

It works.
But what happens if you remove "Ravi"?
Old list:
0  Suresh
1  Ravi
2  Krishna
New list:
0  Suresh
1  Krishna
Notice:
Krishna
Old Index = 2
New Index = 1
Terraform thinks:
"Resource number 1 changed."
Sometimes it destroys and recreates resources because the index changed.
This is the biggest disadvantage of count.
Solution → for_each
Instead of identifying resources by numbers,
Terraform identifies them by names (keys).

Example 1
variable "names" {
  default = [
    "Suresh",
    "Ravi",
    "Krishna"
  ]
}
resource "local_file" "students" {
  for_each = toset(var.names)
  filename = "${each.value}.txt"
  content = "Hello Terraform"
}

Your variable is a list.
[
Suresh
Ravi
Krishna
]
for_each cannot directly use a list.
So we convert it into a set.
{
Suresh
Ravi
Krishna
}
Notice:
No indexes.
Only values.

Terraform now loops like this:

Suresh.txt

Ravi.txt

Krishna.txt
